# 開発環境セットアップ支援アプリ - アイデア精査

## Context

学校の「サービス開発コンテスト」に向けて、開発環境(Claude Code/VSCode, ChatGPT, GitHub)のセットアップを支援するアプリを一から作りたいという相談。当初はPC単体のツールを想定していたが、ヒアリングの結果「スマホアプリとして持ち歩き、現地の未設定PCに対して素早く安全にセットアップを行いたい」という要件が判明。開発は基本ソロ、期間は約1ヶ月あるが学習と並行のため実開発時間は限られる。本ドキュメントは実装前のコンセプト精査(セキュリティ・使い勝手・スコープ)の結果をまとめたもの。

## ヒアリングで確定した前提

- 実行形態: スマホアプリ(vault + セットアップガイド + 各AI/GitHubへの接続)
- 対象PC: 主に「自分専属で使うPC」(SES常駐PC、新規購入PCの初期設定)。ただしチームメンバーが後で同じPCを触る可能性もゼロではない
- 開発体制: ソロ、期間~1ヶ月だが実作業時間は少なめ。機能はかなり絞る前提
- MVPで最低限欲しい機能: ①API管理 ②セットアップ手順表示 ③スマホ自体から各AI(Claude/ChatGPT)への接続 ④GitHubへの接続
- **提出条件: 「動くデモ」ではなく「完成品」であること。かつ配布(デプロイ)までを含める**
  - このアプリはローカル環境で動作するスマホアプリのため、「デプロイ」は主に「誰でもダウンロード・インストールできる状態を作ること」を指すと解釈する
  - この条件により、機能数より「含めた機能を確実に最後まで作り切る」ことを優先する方針にスコープを見直す(下記参照)

## 全体講評

**良い点**
- 環境構築の面倒さは初学者・現場配属時の共通課題で、デモの説得力が出しやすいテーマ
- 「スマホ=個人のセキュアな金庫、PC=使い捨てに近い作業端末」という役割分担は筋が良い(秘密情報の保存先を1箇所に集約できる)

**主な懸念点**
- 当初案の「APIキーをローカルファイルに保存」は、スマホアプリ化する場合もOSの平文ストレージ(SharedPreferences/UserDefaults等)にそのまま保存すると危険。必ずOS標準のセキュアストレージ経由にする
- 「PCにキーを送る」手段を作り込みすぎると、ソロ・限られた開発時間のMVPには収まらない。PC側に専用アプリを作らない設計に倒すのがこの規模感には合っている

## MVP機能とセキュリティ設計

### 1. APIキー/GitHubトークン/環境変数の管理(vault)

「環境変数もセットアップできるようにすべきか」という相談があったが、これは**別機能として新設するのではなく、vaultのデータ構造を汎用化するだけで解決する**ことを推奨する。
- vaultの1エントリを「ラベル(例: `ANTHROPIC_API_KEY`, `GITHUB_TOKEN`, プロジェクト固有の環境変数名など)+値」という汎用の key-value にしておけば、APIキーもGitHubトークンも一般の環境変数も同じ仕組みで扱える
- こうすると「環境変数機能を追加するか」という別判断が不要になり、実装量も増えない(むしろ機能①と分けて作るより実装量は減る)
- ただし「セットアップ」という言葉が「PC側に自動で環境変数を設定する(OSの環境変数やシェルのprofileへの書き込み)」まで含む場合は、それは機能5で決めた通りスコープ外とする。あくまで「値を安全に保管して、コピーしてPC側に手動で貼り付けられる」ところまでがこのアプリの責務
- 保存先はOS標準のセキュアストレージ経由に限定する
  - iOS: Keychain / Android: Keystore(EncryptedSharedPreferences等)
  - React Native/Flutterなら `react-native-keychain` や `flutter_secure_storage` のような既存ライブラリで十分。自前の暗号化コードを書く必要はない
- 平文でのSQLite/JSON保存は不可。ログ出力・クラッシュレポートにもキーが乗らないようマスキングする
- 可能な範囲でキーは用途ごとにスコープを絞ったもの(強い権限の1本のキーを使い回さない)を推奨する一文をアプリ内ヘルプに入れると、審査員へのセキュリティ意識のアピールにもなる

### 2. セットアップ手順の表示
- 静的コンテンツ(自分で書いた手順のMarkdown/JSON)を表示するだけなので技術的リスクは低く、実装コストも小さい。優先度高くMVPに含めて問題ない
- 公式サイトのスクショや文章をそのまま転載しない(著作権・規約面)。自分の言葉で要約したチェックリスト形式が安全かつ実用的

### 3. スマホ本体からのAI接続(Claude/ChatGPT)
- スマホアプリが保存済みキーを使って各社の公式APIに直接リクエストする形になる。通信は必ずHTTPS、キーはvaultから都度読み出してメモリ上でのみ使う(別ストレージへコピーしない)
- レスポンスの会話ログをローカルに保存する場合、そのログ自体も機密度が上がる(コードや設計の相談内容が残るため)。ログもvaultと同等かそれ以上の保護、または保存しない設計を推奨

### 4. GitHubへの接続
- 生のPersonal Access Tokenをユーザーに手入力させるより、GitHubのOAuth Device Flowを使う方が安全かつ体験も良い(ブラウザでコード入力するだけで、長期キーをアプリに直接貼り付けずに済む)
- Device Flowはパブリッククライアントでも実装でき、モバイルアプリとの相性が良い。時間があれば優先的に採用したい項目

### 5. 各AIのトークン/利用状況を確認できるページ(追加希望・要検証)

「簡単に確認できるページが欲しい」という要望自体は妥当だが、**技術的に実現できるかはサービスごとに要検証**なので、次の2段構えを推奨する。

- 前提の注意点: Anthropic/OpenAIとも、個人が発行する通常のAPIキーだけでは「このキーの使用量・残り枠」を返す公開APIが用意されていない/組織管理者向けの別枠APIキー(Admin API key)が必要になるケースが多い。Admin系のキーは通常のAPIキーより強い権限を持つため、vaultに保管する場合はより厳重に扱う必要がある(漏洩時の被害が大きい)
- **MVP版(推奨・確実に完成できる)**: 実際の数値を自前で取得・表示するのではなく、Anthropic Console/OpenAIのUsageページ/GitHubの該当設定ページへの「ショートカット(ディープリンク)」を並べたハブ画面にする。実装は数時間規模で終わり、数値もサービス公式のものなので正確性の心配がない
- **発展版(時間が余れば)**: 各サービスが個人キーでも使用量取得APIを提供している場合に限り、実際の数値をアプリ内に表示する。着手前に各社の公式ドキュメントで「通常のAPIキーで使用量を取得できるか」を必ず確認してから工数を見積もる

### 6. 未設定PCへの受け渡し(今回は「作らない」機能として明示)
- 「完成品として提出」という条件がある以上、中途半端に作って未完成のまま出すことが一番避けたい事態。PC側専用アプリやQRスキャン用HTMLのような「もう1つのアプリ」に相当する機能は、今回は**意図的にスコープ外とする**
- MVPでは、vault画面に表示された値を「コピー」ボタンでコピーし、ユーザーが手動でPC側に入力(貼り付け)する、というシンプルな形に留める。これなら実装・検証・完成の見通しが立てやすい
- QR受け渡し等の仕組みは、機能として面白いのは間違いないため、**完成品を提出した後の発展案**としてアイデアだけ残しておく(今回の開発計画・工数見積もりには含めない)

### 「専属PC」か「共有PCで他人も見る可能性がある」かの切り分けについて

ご相談のあった「モード分け」については、**分けない**ことを推奨する。
- 理由: モードごとに設計・UIが増えると、限られた開発時間で複雑さが跳ね上がる
- 代わりに「PCには常に何も残さない」ことをデフォルト仕様にすれば、専属PCでも共有PCでも同じ設計で安全に倒せる(自分専属PCなら+αで安全、共有PCなら必須要件を満たす、というだけの話になる)
- 具体的には: PC側に鍵を保存するファイルを作らない/作った場合は手順内に「作業後に消す」チェック項目を入れる、という運用ルールをセットアップガイド機能(機能②)に組み込めば十分。これはアプリの構造を増やさずに実現できる

## 技術スタックの方向性(たたき台)

- モバイル: React Native か Flutter(自分の学習状況・使い慣れている言語で選択でよい)。ネイティブのセキュアストレージAPIに素直にアクセスできることと、配布用ビルドの作りやすさを重視する
- PC側: 専用アプリなし。手動コピー&ペーストのみで完結させる(機能5参照)
- バックエンドサーバー: 基本不要(全部ローカル完結の方が「ローカル保存」というコンセプトにも一致し、インフラ運用・デプロイの手間も消せる)

## 配布(デプロイ)計画

「完成品として配布まで含める」を満たすための現実的な着地点は、**Androidに一本化してAPKを配布する**こと。

- OSはAndroidに絞ることを強く推奨
  - iOSは実機配布に Apple Developer Program(年会費)への登録・証明書設定・TestFlight等の審査待ちが発生し、ソロ・短期間の開発では配布だけで時間を持っていかれるリスクが高い
  - Androidは開発者アカウント登録なしで署名付きAPKをビルドでき、審査待ちも発生しない
- 配布方法(いずれもソロで完結できる規模)
  - GitHub Releases にビルド済みAPKを添付し、READMEにダウンロードリンクとインストール手順(提供元不明のアプリの許可設定など)を書く
  - または Firebase App Distribution で審査員に配布用リンクを送る
- 「デプロイまで含める」を成果物ベースで言い換えると、以下がゴールになる
  1. 署名済みAPKがビルドできている
  2. そのAPKがURL経由でダウンロードできる(GitHub Releases等)
  3. 初めて触る人がインストール〜初回起動〜デモ機能一巡までを、自分がその場にいなくても文章の手順だけで完了できる(これは機能②セットアップ手順表示のノウハウをそのまま自分のアプリの配布手順にも転用できる)

## 実装優先順位(時間制約を踏まえて)

「完成品」提出が条件のため、優先順位は**機能を増やす方向ではなく、上から確実に完成させ切る方向**で管理する。時間が足りなくなったら下の項目から丸ごと削る。

1. セットアップ手順表示(静的コンテンツ、最短で完成させやすい)
2. vault(セキュアストレージへの汎用key-value保存・一覧・削除・コピー。APIキー/GitHubトークン/環境変数を共通の仕組みでカバー)
3. GitHub接続(Device Flow)
4. スマホ本体からのAI接続(Claude/ChatGPT呼び出し。UIはシンプルな1問1答形式に留め、ストリーミング表示や会話履歴などは作らない)
5. 各AI利用状況の確認ハブ(まずはショートカットリンク版で確実に完成させる)
6. APKビルド・署名・GitHub Releasesへの公開・README(ダウンロード〜インストール手順)

6番目のデプロイ作業は「機能」ではなく「提出条件」そのものなので、優先順位の最後ではあるが省略不可の必須タスクとして別枠で管理すること。時間が足りなくなった場合は5→4の順で削るか簡素化し、1〜3とデプロイ作業は死守する。

## 未決事項/要検討

- Claude/ChatGPT接続を「公式のログイン(OAuth)」でなく「生のAPIキー」で行うか(サービスによりOAuthが使えない場合は素直にAPIキー+セキュアストレージでよい)
- vault内データのバックアップ/機種変時の引き継ぎ方針(今回は範囲外としてもよいが、審査員から質問が出る可能性はある)
- 各AIサービスが通常のAPIキーで使用量取得APIを提供しているか(機能5の発展版に着手する前に要確認)

デプロイについては「ストアへの正式リリースは不要、ダウンロード可能な状態で可」であることを確認済み。上記の配布計画(Android APK + GitHub Releases)で条件を満たせる。

## 検証方法

- vault: キーがOS平文領域(adb backup/端末のファイルエクスプローラ等)から読めないことを実機で確認
- GitHub Device Flow: 実際にPC側ブラウザでコード入力→認可→トークン取得までの一連の流れを通しで確認
- ログ/クラッシュレポートにAPIキーの値が出力されていないことをコードレビューで確認
- 配布: README記載の手順だけを見て、開発に関わっていない人(先生・友人など)に実際にAPKをダウンロード→インストール→起動してもらい、詰まる箇所がないか確認する
