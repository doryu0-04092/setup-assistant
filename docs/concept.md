# 開発環境セットアップ支援アプリ - コンセプト

## Context

学校の「サービス開発コンテスト」に向けて、開発環境(Cursor / Claude Code / GitHub)のセットアップを支援するアプリを一から作る。開発は基本ソロ、期間は約1ヶ月だが学習と並行のため実開発時間は限られる。

**提出条件: 「動くデモ」ではなく「完成品」であること。かつ配布(デプロイ)までを含める。** この条件により、機能数より「含めた機能を確実に最後まで作り切る」ことを優先する。

## 方針転換の記録(2026-07-20)

当初は次の前提で計画していた。

> スマホ = 個人のセキュアな金庫。自分のAPIキーを暗号化して持ち歩き、現地の未設定PCに流し込む。

この前提でvault(暗号化key-valueストア)まで実装したが、**実際の現場を確認したところ前提が誤っていた**ことが分かった。

- 現場では **会社/常駐先が指定するアカウント・アクセストークン** を使って作業することが多い
- 個人スマホに会社のクレデンシャルを保存するのは、多くの企業のセキュリティポリシー違反になる
- 現地でアカウントが支給されるため、そもそも持ち込む必要がない
- **PCは完全にまっさらとは限らない**。キッティング済み、前任者の設定が残っている、アカウントを切り替えるだけで済む、というケースが実際には多い

よって **vaultを廃止し、「現場のPCで実務環境を立ち上げるための手順ナビゲーター」に一本化する**。

これは機能の後退ではなく設計思想の転換である。「個人端末に秘密情報を残さない」ことは、現場のセキュリティポリシーに沿った正しい設計判断でもある。

## 到達目標(二段階)

| | 状態 | 判定方法 |
|---|---|---|
| 第一段階 | **最低限の実務環境の立ち上げ完了** | clone → 編集 → commit → push が指定アカウントで通る |
| 第二段階 | **自己対処可能な状態** | 以降はAI(Claude Code)とGitHubを使って自分で調べ・解決していける |

品質基準は **「分からない人がこれだけ見て設定できる」** こと。専門用語には説明を添え、どこをクリックするかまで書き、各ステップに成功確認方法を必ず入れる。

## 設計思想: Before / After

| | Before | After |
|---|---|---|
| スマホの役割 | 秘密情報の金庫 | PC画面を占有しない手順書 |
| APIキー | 端末に暗号化保存して持ち歩く | **端末に一切保存しない**。現地で発行しPC側に直接設定 |
| アカウント | 自分のアカウント | 現場指定のアカウント |
| PCの状態 | まっさら前提 | **未導入 / 導入済み の両方に対応** |
| 価値の源泉 | 機能数 | 手順の質(現状確認・分岐・確認方法) |

## 機能構成

### 1. セットアップ手順ナビゲーター(本体)

現場の「まっさらとは限らないPC」を想定した8フェーズ構成。

```
0. 現場ルールの確認     指定アカウントの受領、管理者権限、プロキシ、持ち込み可否
1. 現状の棚卸し         ターミナルの開き方 → 各ツールの有無をコマンドで一括確認
2. Git                  未導入→導入 / 導入済み→アカウント設定の切り替え
3. Node.js              必要な場合のみ
4. Cursor               未導入→導入 / 導入済み→サインアウトして指定アカウントへ
5. Claude Code          Cursor拡張としてインストール、サインイン
6. GitHub連携           認証、PAT発行(スコープ最小)、組織SSO承認、認証情報マネージャ登録
───────────────────────────────────────────────
7. 疎通確認             clone→編集→commit→push が通る = 【立ち上げ完了】
8. 自走準備             AIに聞ける・GitHubで調べられる状態にする = 【自己対処可能】
```

**各フェーズの冒頭に「現状確認」を置く**のがこのアプリの肝。「入っているか」「どのアカウントで入っているか」をまず確認し、その結果で「未導入向け」「導入済み向け(切り替えのみ)」に手順が分岐する。利用者には該当する手順だけを表示し、不要な手順を見せない。

### 2. ターミナル操作とUI操作の明示

分からない人にとって「どこで何をするのか」が最大の詰まりポイントになるため、すべてのステップと確認方法に操作面を持たせ、視覚的に区別する。

| | ターミナル操作 | UI操作 |
|---|---|---|
| 表示 | 等幅フォントの黒地コマンドブロック + ターミナルアイコン | 通常テキスト + 画面アイコン |
| 内容 | 実行するコマンド(コピーボタン付き)+ 期待される出力例 | クリック経路を「〜を開く → 〜をクリック」形式で |
| 確認 | コマンドの出力で判定 | 画面上の表示で判定 |

ターミナルはWindows前提のため **PowerShell** を基準にし、開き方自体をフェーズ1で説明する(ターミナルを触ったことがない人がいる前提)。

手順に必ず含める主要コマンド:

| 目的 | コマンド |
|---|---|
| Gitの有無・バージョン確認 | `git --version` |
| Git設定の現状確認 | `git config --global --list` |
| Gitユーザー設定(指定アカウントへ切り替え) | `git config --global user.name` / `user.email` |
| Node.jsの有無確認 | `node --version` / `npm --version` |
| GitHub CLIの有無確認 | `gh --version` |
| GitHub認証状態の確認(どのアカウントで入っているか) | `gh auth status` |
| GitHubへログイン / 切り替え | `gh auth login` / `gh auth logout` |
| 疎通確認 | `git clone` → `git add` → `git commit` → `git push` |
| リモート設定の確認 | `git remote -v` |

### 3. 進捗の保存

チェック状態を端末に保存し、作業を中断しても「どこまでやったか」が残るようにする。現場では1日で終わらないことも多い。チェック状態は秘密情報ではないため通常のDataStoreでよい。

### 4. メモと手順の上書き編集

現場ごとに手順は微妙に違う(社内リポジトリのURL、プロキシ設定、独自ルール)ため、利用者が自分用に手を入れられるようにする。

- **メモ**: 全ステップに自由記述のメモを付けられる。後から変更・確認できる
- **手順の上書き**: ステップの説明文とコマンドを自分用に書き換えられる。上書きしたステップには「編集済み」バッジを出し、元に戻す操作を用意する
- 上書きしても `lastVerified`(公式で最終確認した日)は元のまま表示し、「アプリ標準の手順」と「自分が書き換えた手順」を混同させない

### 5. 安全な進め方(説明画面)

なぜ端末に秘密情報を保存しないのか、現場PCでの取り扱い原則、作業後のクリーンアップ項目を説明する。

## 秘密情報の扱い

**このアプリはAPIキーやトークンを保存しない。** 現地で発行したものは、その場でPC側(環境変数・認証情報マネージャ)に直接設定する。

ただしメモ欄に利用者がうっかりトークンを書く可能性があるため、**メモと上書き内容は暗号化して保存する**(Android Keystore / `security-crypto`)。

- これは「秘密情報を保存する機能」ではなく、**うっかり書かれた場合に平文で残さないための保険**
- 入力欄には「トークンやパスワードは書かないでください」と明示する
- vault(保存を目的とした機能)は廃止したまま。目的が違う

## 削除した機能とその理由

| 機能 | 削除理由 |
|---|---|
| vault(暗号化key-valueストア) | 現場では指定アカウントを使うため持ち込む必要がなく、個人端末への保存自体がポリシー違反になりうる |
| GitHub Device Flow連携 | 「自分のアカウントと連携する」前提だった。現場では指定アカウントを使うため成立しない |
| スマホ本体からのAI接続 | 「自分のAPIキーで叩く」前提だった。同上 |
| AI利用状況の確認ハブ | 単独機能としては優先度が低い。必要なリンクは手順内に吸収する |

手順の陳腐化対策として導入した **`lastVerified`(公式サイトで最終確認した日)の表示** と、**公式サイトへのリンク** は引き続き有効な仕組みとして残す。要約が古くなっても、利用者は常に一次情報へ辿れる。

## 技術スタック

- **Android(Kotlin + Jetpack Compose)一本化**。iOS配布はApple Developer Program登録・審査が必要で、ソロ・短期間の開発では配布だけで時間を持っていかれるため見送る
- バックエンドサーバーなし。全てローカル完結
- PC側の専用アプリなし

## 配布計画

「完成品として配布まで含める」を満たす着地点は **署名付きAPKをGitHub Releasesで配布** すること。Androidは開発者アカウント登録なしで署名付きAPKをビルドでき、審査待ちも発生しない。

成果物ベースのゴール:
1. 署名済みAPKがビルドできている
2. そのAPKがURL経由でダウンロードできる(GitHub Releases)
3. 初めて触る人が、インストール〜初回起動〜機能一巡までを文章の手順だけで完了できる

なお、審査員がインストールせずブラウザから触れる形も検討中(Appetize.io等のブラウザ内エミュレータ配信)。無料枠の時間制限が課題のため未確定。

## 品質保証の仕組み

開発機にAndroid SDKを入れていないため、**GitHub Actions上での実ビルドが唯一の動的確認手段**として機能している。

- `android-build`: pushごとに `gradle assembleDebug` を実行。実際にこれで2件のコンパイルエラーを検出・修正した
- `link-check`: 手順内の公式URLの死活監視(週次 + 変更時)。Bot対策の403を誤検知しないよう、404/410/5xxのみ失敗扱いにしている

## 実装優先順位

上から確実に完成させ切る。時間が足りなくなったら下から丸ごと削る。

1. セットアップ手順ナビゲーター(現状確認の分岐、ターミナル/UIの区別、進捗保存)
2. 安全な進め方の説明画面
3. メモと手順の上書き編集
4. APKビルド・署名・GitHub Releasesへの公開・README整備

4番目のデプロイ作業は「機能」ではなく「提出条件」そのものなので、優先順位の最後ではあるが省略不可の必須タスクとして別枠で管理する。

## 検証方法

- 手順の内容: 実際に公式ドキュメントを確認しながら作成し、作成後にレビューする(現場の実態と合っているか、分からない人が読める水準か)
- 進捗・メモの永続化: アプリを再起動しても保持されていることを実機/エミュレータで確認
- ログ/クラッシュレポートにメモの内容が出力されていないことをコードレビューで確認
- 配布: READMEの手順だけを見て、開発に関わっていない人に実際にAPKをダウンロード→インストール→起動してもらい、詰まる箇所がないか確認する
